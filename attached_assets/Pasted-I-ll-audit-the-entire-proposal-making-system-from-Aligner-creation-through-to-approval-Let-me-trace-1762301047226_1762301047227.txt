Audit Findings
1. Aligner Chat → Proposal Creation Flow
Issue Found: The "Agree and create proposals" flow has redundant complexity.

Currently there are TWO endpoints that create proposals from Aligner chat:

/api/aligner/agree-and-create-proposals - Sends confirmation message, gets JSON, creates proposals
/api/aligner/create-proposals-from-chat - Extracts JSON from existing messages, creates proposals
Problem: Both endpoints essentially do the same thing but with different triggers. The agree-and-create-proposals endpoint sends a user message "Yes, I agree" then immediately tries to parse JSON from the response, which adds unnecessary complexity.

Recommendation: Consolidate into a single flow where users explicitly click "Create Proposals" after reviewing Aligner's suggested edits.

2. JSON Edit Format Validation
Issue Found: Inconsistent validation of the edits JSON structure.

In proposal-diff-viewer.tsx:

const isValidEdits = edits.length > 0 && edits.every(e => 
  'old' in e && 'new' in e && 'reason' in e && e.new && e.reason

This allows empty old field (for new content additions) but requires non-empty new and reason.

However, in the backend approval flow (/api/kb/proposals/:id/approve), the edit application logic expects:

  // New content addition
} else if (edit.old) {
  // Replacement
}
Problem: The frontend validation doesn't check if old is at least defined, just that it exists as a key.

Fix Needed: Add visual feedback to show which edits have previously failed to apply.

Issue Found: When user approves only some edits, the system:
