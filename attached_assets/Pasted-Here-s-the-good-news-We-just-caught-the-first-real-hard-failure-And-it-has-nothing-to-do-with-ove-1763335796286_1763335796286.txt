What this will reveal

The invalid Date ALWAYS comes from one of these three places:
	1.	recipientOpenLocal computed like:

new Date(formatInTimeZone(...))

	2.	recipientEndLocal computed like:

new Date(formatInTimeZone(...))

	3.	A timezone string becoming something like:
Europe/WarsawT... or
America/New_YorkTundefined:00:00

Right now, we already see:
	•	Recipient is Maine
	•	Maine → America/New_York
	•	Opening time Monday: 11am
→ 660 minutes
→ openHour = 11
→ openMinute = 0
Everything looks valid.

But the crash proves:

Either openHour, openMinute, closeHour, or closeMinute is NaN.

Or:

formatInTimeZone returned an invalid string (this happens when dayLoopDate is malformed).

Given the logs:

dayLoopDate (raw UTC): 2025-11-16T23:28:12.964Z

and:

Admin UTC Window: 2025-11-17T15:00:00.000Z → 2025-11-17T23:00:00.000Z

Admin window is valid.

The break must be in:
	•	recipientOpenLocal
	•	recipientEndLocal
