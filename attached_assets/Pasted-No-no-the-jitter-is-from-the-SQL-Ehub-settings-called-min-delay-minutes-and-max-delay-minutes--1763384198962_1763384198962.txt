Looking at the codebase, here are the functions that read from or write to the ehub_settings SQL table:

Database Layer (storage.ts)
getEhubSettings() - Reads settings from the table
updateEhubSettings(updates) - Updates or creates settings in the table
normalizeEhubSettings(settings) - Helper that converts PostgreSQL decimal types to numbers
Service Layer
Email Queue System:

server/services/emailQueue.ts - processEmailQueue() calls getEhubSettings() to check sending hours
server/services/Matrix2/slotGenerator.ts - ensureDailySlots() calls getEhubSettings() to read all slot generation parameters (dailyEmailLimit, sendingHoursStart/End, min/maxDelayMinutes)
Email Scheduling:

server/services/emailSchedulingService.ts - scheduleRecipient() calls getEhubSettings() to read timing parameters
server/services/eligibilityPlanner.ts - calculateNextEligibleSendTime() calls getEhubSettings() for client window settings (clientWindowStartOffset, clientWindowEndHour)
Reply Detection:

server/services/gmailReplyDetection.ts - Doesn't currently call it but uses settings indirectly
API Routes (routes.ts)
GET /api/ehub/settings - Returns settings via getEhubSettings()
PATCH /api/ehub/settings - Updates settings via updateEhubSettings()
Current Settings Schema
The ehub_settings table stores:

minDelayMinutes (jitter min)
maxDelayMinutes (jitter max)
jitterPercentage (variance %)
dailyEmailLimit (max emails/day)
sendingHoursStart (company start hour)
sendingHoursEnd (company end hour)
clientWindowStartOffset (hours after business opens)
clientWindowEndHour (client local cutoff)
promptInjection (AI instructions)
keywordBin (AI context keywords)
skipWeekends (boolean)
All the slot generation, scheduling, and queue processing functions depend on these settings.
