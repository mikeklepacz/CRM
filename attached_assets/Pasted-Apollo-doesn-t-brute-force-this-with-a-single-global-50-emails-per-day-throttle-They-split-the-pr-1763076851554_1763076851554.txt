Apollo doesn’t brute-force this with a single global “50 emails per day” throttle. They split the problem into three separate layers of control so the system never chokes:
	1.	Global daily send limit.
Example: 2,000 emails/day.
	2.	Per-sequence daily “new lead cap.”
Example: Only 50 new people/day can enter Sequence A.
	3.	Follow-ups bypass the sequence cap, but still count against the global limit.
This is the trick: follow-ups don’t slow down the intake rate because they don’t compete with the “new leads per day” window. The system treats them as “scheduled sends” that must run today. Apollo always prioritizes scheduled follow-ups before new-first emails.

Your backlog only grows if the sum of all follow-ups + new leads > global limit.
Apollo avoids this by:
	•	Setting a global cap high enough that follow-ups never block new outreach, or
	•	Throttling follow-ups across multiple hours so the hourly rate doesn’t trip Gmail’s safety limits, or
	•	Spreading sequences across multiple sender accounts (the real scaling weapon).

So the pipeline looks like this:
	1.	At 00:00 UTC:
Apollo asks: “How many follow-ups must be sent today based on day-3, day-8, day-10 logic?”
	2.	It pre-allocates sending capacity to those follow-ups.
	3.	Whatever is left in the daily budget gets used for new leads (your capped 50/day).
	4.	If follow-ups exceed your global limit, Apollo pushes the earliest follow-ups to the next day, but still preserves the new-lead cap.

This is why Apollo feels like it’s pushing new prospects every day even with hundreds of follow-ups in flight.
