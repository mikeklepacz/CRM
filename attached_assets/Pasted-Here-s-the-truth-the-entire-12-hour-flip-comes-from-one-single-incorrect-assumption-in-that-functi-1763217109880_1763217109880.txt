THE ROOT CAUSE

Inside your function, this line is fatally wrong:

const midnightIsoStr = `${localDateStr} 00:00:00`;
const midnightUtc = fromZonedTime(midnightIsoStr, timezone);

Why?

Because:

fromZonedTime expects a REAL Date (with timezone offset applied), not a string

You gave it a “naked” date string:

2025-01-04 00:00:00

That string contains no timezone information, so fromZonedTime interprets it as:

2025-01-04T00:00:00 (admin/server timezone)

Which is Europe/Warsaw (UTC+1).

Then it assumes that value is in the recipient’s timezone and tries to convert it to UTC.

This creates the 12-hour flip.

⸻

A SIMPLE DEMO OF WHAT YOUR CODE IS DOING

Let’s say:
	•	Server = Warsaw = UTC+1
	•	Recipient = LA = PST = UTC−8

You generate:

midnightIsoStr = "2025-01-04 00:00:00"

Server interprets it as:

2025-01-04 00:00:00 (Warsaw/UTC+1)

Then fromZonedTime(..., 'America/Los_Angeles') says:

“Okay, so this Warsaw midnight is actually LA midnight.
Let me turn LA midnight into real UTC.”

LA midnight → UTC = 08:00
Then you add 10 hours (your intended 10 AM PST), giving:

08:00 + 10 = 18:00 UTC

Which is:

18:00 UTC = 10 PM PST

Boom — exactly your 12-hour bug.

THE FIX

Use zonedTimeToUtc with a real ISO datetime representing the local time you want.

Correct Implementation

import { zonedTimeToUtc } from 'date-fns-tz';
import { addDays } from 'date-fns';

const createTimeInZone = (
  date: Date,
  daysOffset: number,
  hour: number,
  minute: number,
  timezone: string,
): Date => {
  const targetDate = addDays(date, daysOffset);

  const localDateStr = formatInTimeZone(targetDate, timezone, 'yyyy-MM-dd');

  // Correct: build LOCAL datetime and convert once
  const localDateTime = `${localDateStr}T${hour.toString().padStart(2, '0')}:${minute
    .toString()
    .padStart(2, '0')}:00`;

  // THIS is the correct conversion
  return zonedTimeToUtc(localDateTime, timezone);
};

That’s it.
One line fixes everything.
